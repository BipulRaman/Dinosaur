Built for files that break other apps

Everything is bounded by what's on screen — not by how big the file is.

📂

Instant open

Memory-mapped I/O means a 170 GB file opens as fast as a 50 KB one. Nothing is read into RAM until you scroll to it.

📊

Spreadsheet grid

Clean, Google-Sheets-like typography with crisp gridlines, row numbers and sticky headers.

🖱️

Select & copy

Click or drag to select a range of cells, then right-click or press Ctrl+C to copy as TSV.

🔎

Fast search

Find text forwards or backwards across the entire file, with optional case sensitivity.

🔢

Go to any row

Jump to row 400-million in a billion-row file in O(1) — no scrolling, no waiting.

🖥️

Runs anywhere

Ships a CPU software renderer, so it works on cloud VMs and machines without a dedicated GPU.

Why it stays fast

A few deliberate techniques keep memory and CPU bounded by the viewport.

Memory-mapped I/O
The OS pages in only the bytes you touch. The whole file is never in RAM.
Sparse parallel index
Newlines are scanned across all cores; one offset is stored every 4096 lines, so 500M lines cost a few MB — and any line is found in O(1).
Virtualized table
Only the ~50 rows on screen are ever drawn.
Integer virtual scroll
Scrolling tracks the row index, not pixels, so it stays precise across billions of rows.
Background indexing
Indexing runs on a worker thread with a live progress bar — the UI never freezes.

Formats it speaks

.csvComma-separated values
.tsvTab-separated values
.ndjsonJSON Lines (NDJSON) — one JSON value per line; first object's keys become columns
.txt / .logOne line per row

JSON must be newline-delimited (NDJSON / JSON Lines) — every line a complete JSON value. Nested values show as compact JSON text in the cell; columns come from the first line. A single pretty-printed JSON array/object spanning many lines isn't streamed — convert it first, e.g. jq -c '.[]' big.json > big.ndjson.
